1 Visão Geral do Projeto

Antes de criarmos a primeira rota de uma API, precisamos montar a infraestrutura que sustenta tudo. Pense nisso como construir uma casa: você não coloca os móveis antes de ter paredes, encanamento e elétrica.

Neste tutorial vamos dissecar o que foi construído no projeto inicial, entender por que cada peça existe e como elas se conectam — para que quando você criar suas próprias entidades, saiba exatamente onde colocar cada coisa.

Camadas da Arquitetura

A API segue uma arquitetura em camadas. Cada camada tem uma responsabilidade única e só se comunica com a camada abaixo dela:

🌐 Routes — recebe a requisição HTTP, chama o Service, retorna a resposta AuthRoutes, UsuarioRoutes…
⚙️ Services — regras de negócio (o que a aplicação faz) AuthService, UsuarioService…
🗄️ Repositories — acesso ao banco de dados UsuarioRepositoryImpl
🍃 MongoDB — persistência dos dados Collection<Usuario>
💡 Por que separar em camadas? Se amanhã precisarmos trocar o MongoDB por PostgreSQL, só mudamos a camada Repository. As rotas e os services não precisam saber onde os dados são armazenados. Isso é separação de responsabilidades.

2 Application.kt — O Ponto de Entrada

Este é o arquivo que o Gradle executa quando você roda ./gradlew run. Ele é responsável por ligar todas as peças do sistema. 

Application.ktfun main() {
    val port = System.getenv("PORT")?.toIntOrNull() ?: 7351
    embeddedServer(Netty, port = port, module = Application::module).start(wait = true)
}

fun Application.module() {

    // 1. Injeção de dependência
    install(Koin) {
        modules(appModule)
    }

    val jwtConfig = JwtConfig()

    // 2. Plugins
    configureContentNegotiation()
    configureCORS()
    configureAuthentication(jwtConfig)
    configureStatusPages()
    configureSwagger()

    // 3. Rotas
    routing {
        authRoutes()
        usuarioRoutes()
        perfilRoutes()
        specialRoutes()
    }
}

Ponto de Entrada (main)

A função main() cria um servidor Netty na porta configurada:

  • Primeiro tenta ler a variável de ambiente PORT (útil em produção/Docker).
  • Se não existir, usa a porta 7351 como padrão.
  • embeddedServer cria o servidor em memória — sem precisar de um Tomcat ou servidor externo.
  • wait = true faz a thread principal bloquear (o servidor fica rodando até você encerrar com Ctrl+C).

Função module()

O module() é uma função de extensão sobre Application — ele "configura" a aplicação Ktor. A ordem importa:

  1. Koin primeiro — precisa estar instalado antes de qualquer código que tente injetar dependências.
  2. Plugins depois — JSON, CORS, Auth são registrados globalmente.
  3. Rotas por último — são o "topo" da pirâmide; dependem de tudo que veio antes.

3 Model — A Representação dos Dados

O model define como um Usuário existe no sistema. É a classe que será salva no MongoDB e trafegada entre as camadas da aplicação.

data class Usuario

models/Usuario.kt@Serializable
data class Usuario(
    val id: String? = null,
    val nome: String,
    val email: String,
    val senhaHash: String,            // nunca a senha em texto!
    val papeis: Set<Papel>,
    val instituicaoId: String? = null,
    val ativo: Boolean = true,
    val avatar: String? = null,
    val fusoHorario: String = "America/Manaus",
    val accesstoken: String? = null,
    val refreshtoken: String? = null,
    val tokenUnico: String? = null,
    val codigoRecuperaSenha: String? = null,
    @Contextual val expCodigoRecuperaSenha: Instant? = null,
    @Contextual val ultimoLoginEm: Instant? = null,
    @Contextual val createdAt: Instant = Instant.now(),
    @Contextual val updatedAt: Instant? = null
)

Por que data class?

data class gera automaticamente equals(), hashCode(), toString() e copy(), que são essenciais para comparar e clonar objetos. No contexto de repositórios, isso facilita muito os testes.

Por que id é nullable (String?)?

Quando criamos um usuário novo (antes de salvar), ele ainda não tem ID — por isso null. O ID é atribuído com UUID.randomUUID().toString() no Service antes de chamar insert(). Se fosse não-nullable, seria obrigatório fornecer um ID na construção, o que não faz sentido.

Por que guardar accesstoken e refreshtoken no banco?

Para conseguir invalidar sessões: se um usuário fizer logout ou mudar a senha, apagamos o token do banco. Na próxima requisição, mesmo que o token ainda não tenha expirado pelo tempo, ele não bate mais com o banco — e é rejeitado. Isso implementa o conceito de revogação de token.

enum class Papel

models/Usuario.ktenum class Papel {
    ADMIN_PLATAFORMA,   // visão global, gerencia tudo
    ADMIN_INSTITUICAO,  // administrador de uma instituição
    OPERADOR,           // atua no atendimento
    USUARIO_FINAL       // cidadão comum
}

Usar enum em vez de String garante que nenhum papel inválido entre na aplicação. Se alguém tentar criar um usuário com papel "super_hacker", o Kotlin jogará uma exceção na desserialização.

@Serializable e @Contextual

AnotaçãoO que fazQuando usar
@Serializable Diz ao compilador para gerar código de serialização/desserialização JSON automaticamente Em toda data class ou enum que trafega pela API ou é salvo no banco
@Contextual Delega a serialização daquele campo ao serializer contextual registrado (no nosso caso, InstantSerializer) Em campos de tipos que o kotlinx.serialization não sabe serializar nativamente, como java.time.Instant
⚠️ Atenção: campos sensíveis na resposta A classe Usuario contém senhaHash, accesstoken e refreshtoken. Esses campos nunca devem aparecer na resposta da API. Por isso usamos DTOs de resposta (como PerfilResponse) que incluem apenas os campos públicos.

4 MongoConfig — Conexão com o Banco

Conexão com KMongo

config/MongoConfig.ktclass MongoConfig(
    connectionString: String = System.getenv("MONGO_URI") ?: "mongodb+srv://...",
    databaseName: String = System.getenv("MONGO_DB") ?: "Aula"
) {
    private val client = KMongo.createClient(connectionString)
    val database: MongoDatabase = client.getDatabase(databaseName)

    val usuarios get() = database.getCollection<Usuario>(Constants.COLLECTION_USUARIOS)
}

KMongo.createClient() cria uma conexão com o MongoDB. O cliente é criado uma única vez (registrado como single no Koin) e reutilizado em todas as requisições.

Coleções tipadas

database.getCollection<Usuario>("usuarios") retorna um objeto MongoCollection<Usuario>. Isso significa que quando você fizer collection.findOne(...), o resultado já virá desserializado como um objeto Usuario — sem precisar converter manualmente de JSON.

ℹ️ Uma coleção por entidade Para cada nova entidade que você criar (ex: Produto), adicione uma propriedade correspondente no MongoConfig: val produtos get() = database.getCollection<Produto>("produtos")

5 Injeção de Dependência com Koin

O que é Injeção de Dependência?

Imagine que o AuthService precisa do UsuarioRepository para funcionar. Sem DI, o AuthService criaria o repositório dentro de si: val repo = UsuarioRepositoryImpl(...). O problema: o service ficaria acoplado à implementação concreta — impossível de testar e difícil de trocar.

Com Injeção de Dependência, alguém externo (o Koin) cria as dependências e as "injeta" no service. O service declara o que precisa, mas não sabe como é criado.

O appModule

config/KoinModules.ktval appModule = module {

    // Config
    single { MongoConfig() }
    single { JwtConfig() }

    // Repositories — injeta a coleção tipada do MongoConfig
    single<UsuarioRepository> { UsuarioRepositoryImpl(get<MongoConfig>().usuarios) }

    // Services auxiliares
    single { FileStorageService() }
    single { EmailService() }

    // Services principais — get() resolve a dependência automaticamente
    single { AuthService(get(), get(), get()) }
    single { UsuarioService(get(), get()) }
    single { PerfilService(get()) }
}

single vs factory

DeclaraçãoComportamentoQuando usar
single { ... } Cria uma única instância compartilhada em toda a aplicação (Singleton) Conexões de banco, configs, services sem estado mutável por requisição
factory { ... } Cria uma nova instância a cada injeção Objetos que precisam ser isolados por requisição ou teste

A função get() dentro do módulo é o Koin resolvendo uma dependência registrada. No AuthService(get(), get(), get()), o Koin injeta automaticamente UsuarioRepository, JwtConfig e EmailService — na ordem dos parâmetros do construtor.

💡 Para usar um Service numa rota, faça assim: val authService: AuthService by inject() — o Koin encontra e entrega a instância registrada em appModule.

6 JwtConfig — Autenticação com Tokens

Como funciona o JWT

JWT (JSON Web Token) é um token compacto e auto-suficiente. Ele contém claims (informações) sobre o usuário assinadas com uma chave secreta. Como a assinatura é verificável, qualquer servidor com a mesma chave pode validar o token sem consultar o banco de dados a cada requisição.

Access Token e Refresh Token

config/JwtConfig.kt// Access Token — curta duração (15 minutos)
fun generateAccessToken(userId: String, papeis: Set<Papel>, instituicaoId: String?): String {
    return JWT.create()
        .withClaim("id", userId)
        .withClaim("papeis", papeis.map { it.name })
        .withClaim("instituicaoId", instituicaoId)
        .withExpiresAt(Date(System.currentTimeMillis() + Constants.JWT_ACCESS_EXPIRATION_MS))
        .sign(algorithm)
}

// Refresh Token — longa duração (7 dias)
fun generateRefreshToken(userId: String): String {
    return JWT.create()
        .withClaim("id", userId)
        .withClaim("type", "refresh")
        .withExpiresAt(Date(System.currentTimeMillis() + Constants.JWT_REFRESH_EXPIRATION_MS))
        .sign(algorithm)
}
TokenValidadeContémQuando usar
Access Token 15 minutos id, papeis, instituicaoId Enviado em toda requisição protegida no header Authorization
Refresh Token 7 dias id, type="refresh" Enviado ao endpoint POST /auth/refresh para obter um novo access token

7 Repository — Acesso ao Banco

Interface — o Contrato

O padrão Repository separa a lógica de acesso a dados do resto da aplicação. A interface define o contrato (o que pode ser feito), sem dizer como: 

repository/UsuarioRepository.ktinterface UsuarioRepository {
    fun findById(id: String): Usuario?
    fun findByEmail(email: String): Usuario?
    fun findAll(page: Int, limit: Int, filters: Map<String, Any?>): Pair<List<Usuario>, Long>
    fun insert(usuario: Usuario): Usuario
    fun update(id: String, updates: Map<String, Any?>): Boolean
    fun delete(id: String): Boolean
    fun findByTokenUnico(token: String): Usuario?
    fun findByCodigoRecuperacao(codigo: String): Usuario?
}

O retorno de findById é Usuario? — nullable. Isso força quem chama a tratar o caso em que o usuário não existe, em vez de explodir com um NullPointerException.

Implementação com KMongo

repository/UsuarioRepositoryImpl.ktclass UsuarioRepositoryImpl(
    private val collection: MongoCollection<Usuario>
) : UsuarioRepository {

    override fun findById(id: String) = collection.findOneById(id)

    override fun findByEmail(email: String) =
        collection.findOne(Usuario::email eq email.lowercase())

    override fun insert(usuario: Usuario): Usuario {
        collection.insertOne(usuario)
        return usuario
    }

    override fun update(id: String, updates: Map<String, Any?>): Boolean {
        val setUpdates = updates.map { (key, value) -> Updates.set(key, value) }
        val result = collection.updateOneById(id, combine(setUpdates))
        return result.modifiedCount > 0
    }
}

Perceba que o KMongo permite usar a referência Kotlin do campo com Usuario::email eq email em vez de strings mágicas como "email" to email. Isso evita erros de digitação no nome do campo.

Filtros Dinâmicos

O método buildFilters constrói a query MongoDB a partir de um mapa de critérios. Isso permite que a mesma listagem suporte vários filtros opcionais sem duplicar código: 

private fun buildFilters(filters: Map<String, Any?>): List<Bson> {
    return filters.mapNotNull { (key, value) ->
        when {
            value == null         -> null          // ignora campo ausente
            key == "nome"  -> Usuario::nome  regex Regex(".*$value.*", ...)
            key == "email" -> Usuario::email regex Regex(".*$value.*", ...)
            key == "ativo" -> Usuario::ativo eq (value as Boolean)
            else                   -> null
        }
    }
}

8 Services — As Regras de Negócio

Os Services são o coração da aplicação. Eles não conhecem HTTP (nada de call.respond() aqui) e não conhecem o banco diretamente (usam o Repository). Sua única preocupação é a lógica de negócio.

AuthService

services/AuthService.ktclass AuthService(
    private val usuarioRepository: UsuarioRepository,  // interface, não a impl!
    private val jwtConfig: JwtConfig,
    private val emailService: EmailService
) {
    fun login(request: LoginRequest): TokenResponse {
        val usuario = usuarioRepository.findByEmail(request.email)
            ?: throw ApiException(401, "Credenciais inválidas")

        if (!usuario.ativo) throw ApiException(403, "Usuário desativado")

        if (!PasswordUtils.verifyPassword(request.senha, usuario.senhaHash))
            throw ApiException(401, "Credenciais inválidas")

        val accessToken = jwtConfig.generateAccessToken(...)
        val refreshToken = jwtConfig.generateRefreshToken(...)

        // Salva os tokens no banco para controle de sessão
        usuarioRepository.update(usuario.id!!, mapOf(
            "accesstoken" to accessToken,
            "refreshtoken" to refreshToken,
            "ultimoLoginEm" to Instant.now()
        ))

        return TokenResponse(accessToken, refreshToken, jwtConfig.accessTokenExpiresAt())
    }
}

Perceba como as regras de negócio ficam claras:

  1. Busca o usuário pelo e-mail.
  2. Verifica se ele existe — se não, lança ApiException(401).
  3. Verifica se está ativo.
  4. Verifica a senha com BCrypt.
  5. Gera os tokens e salva no banco.
ℹ️ Por que usamos a mesma mensagem para "usuário não encontrado" e "senha errada"? Para não revelar ao atacante se o e-mail existe no sistema. Se a mensagem fosse "e-mail não cadastrado", um atacante poderia enumerar quais e-mails existem no banco.

UsuarioService

Lida com as operações CRUD de usuários: listar, buscar por ID, atualizar, deletar e criar via instituição.

PerfilService

Lida com o "meu perfil" — editar nome/e-mail, trocar senha, fazer upload/remoção de avatar.

9 Plugins Ktor

Plugins são extensões instaladas na aplicação Ktor via install(). Eles interceptam todas as requisições e respostas, adicionando comportamentos globais.

ContentNegotiation — Serialização JSON

plugins/ContentNegotiation.ktfun Application.configureContentNegotiation() {
    install(ContentNegotiation) {
        json(Json {
            prettyPrint = true         // JSON indentado (mais legível)
            isLenient = true            // tolera JSON levemente mal-formado
            ignoreUnknownKeys = true    // ignora campos extras no request
            encodeDefaults = true       // inclui campos com valor default
            serializersModule = SerializersModule {
                contextual(InstantSerializer)  // registra serializer de Instant
            }
        })
    }
}

Sem este plugin, o Ktor não saberia converter objetos Kotlin para JSON e vice-versa. ignoreUnknownKeys = true é especialmente útil quando o cliente envia campos extras (como campos futuros) — a API não quebra.

CORS

plugins/CORS.ktinstall(CORS) {
    anyHost()                              // permite qualquer origem
    allowHeader(HttpHeaders.Authorization)  // permite o header JWT
    allowMethod(HttpMethod.Get)
    allowMethod(HttpMethod.Post)
    allowMethod(HttpMethod.Patch)
    allowMethod(HttpMethod.Delete)
}

CORS (Cross-Origin Resource Sharing) é uma política de segurança dos navegadores. Sem este plugin, qualquer frontend hospedado em domínio diferente da API seria bloqueado pelo browser.

⚠️ anyHost() é para desenvolvimento! Em produção, substitua por allowHost("meuapp.com") para permitir apenas origens conhecidas.

Authentication — Validação JWT

plugins/Authentication.ktfun Application.configureAuthentication(jwtConfig: JwtConfig) {
    install(Authentication) {
        jwt("auth-jwt") {
            realm = jwtConfig.realm
            verifier(jwtConfig.verifier)   // valida a assinatura
            validate { credential ->
                val userId = credential.payload.getClaim("id").asString()
                if (userId != null) JWTPrincipal(credential.payload)
                else null
            }
            challenge { _, _ ->
                call.respond(Unauthorized, ApiResponse.error<Any>("Token inválido ou ausente"))
            }
        }
    }
}

Este plugin registra o esquema de autenticação "auth-jwt". Para proteger uma rota, você adiciona authenticate("auth-jwt") { ... } ao redor dela. O Ktor verifica o token automaticamente — se inválido, a função challenge responde com 401.

Authorization — Controle de Papéis

plugins/Authorization.ktfun Route.authorize(vararg papeis: Papel, build: Route.() -> Unit): Route {
    val route = createChild(AuthorizationRouteSelector())

    route.intercept(ApplicationCallPipeline.Call) {
        val userPapeis = call.currentUserPapeis()
        if (userPapeis.none { it in papeis }) {
            call.respond(Forbidden, ApiResponse.error<Any>("Acesso negado"))
            finish()
        }
    }

    route.build()
    return route
}

Este é um plugin customizado (não é nativo do Ktor). Ele cria uma camada de autorização baseada em papéis. Uso nas rotas: 

authenticate("auth-jwt") {
    authorize(Papel.ADMIN_PLATAFORMA) {
        get("/admin/dashboard") { /* apenas admins chegam aqui */ }
    }
}
💡 Diferença entre Authentication e Authorization:
Authentication (Autenticação) = "quem é você?" — verifica o token JWT.
Authorization (Autorização) = "o que você pode fazer?" — verifica os papéis.

StatusPages — Tratamento Centralizado de Erros

plugins/StatusPages.ktinstall(StatusPages) {
    exception<ApiException> { call, cause ->
        call.respond(
            HttpStatusCode.fromValue(cause.statusCode),
            ApiResponse.error<Any>(cause.message, cause.errors)
        )
    }
    exception<Exception> { call, cause ->
        call.respond(InternalServerError, ApiResponse.error<Any>("Erro interno"))
    }
}

Sem este plugin, qualquer exceção não tratada derrubaria a requisição com uma resposta em branco. Com o StatusPages, qualquer throw ApiException(404, "Não encontrado") lançado em qualquer camada é capturado aqui e retorna um JSON padronizado.

ApiException é nossa exceção customizada: 

plugins/ApiException.ktclass ApiException(
    val statusCode: Int,
    override val message: String,
    val errors: List<String>? = null
) : RuntimeException(message)

Swagger — Documentação Interativa

O plugin Swagger serve uma interface visual para testar todos os endpoints da API acessando http://localhost:7351/api-docs. A documentação é gerada a partir do arquivo src/main/resources/openapi/documentation.json.

10 DTOs — Objetos de Transferência de Dados

DTO (Data Transfer Object) é uma classe que define exatamente quais campos entram ou saem pela API em cada endpoint específico. É a camada de contrato entre o cliente e a API.

Requests

dtos/request/AuthRequests.kt@Serializable
data class LoginRequest(
    val email: String,
    val senha: String
)

@Serializable
data class RegisterRequest(
    val nome: String,
    val email: String,
    val senha: String
)

O cliente HTTP envia um JSON com esses campos. O Ktor desserializa para a data class via call.receive<LoginRequest>() na rota.

Responses e ApiResponse

dtos/response/ApiResponse.kt@Serializable
data class ApiResponse<T>(
    val success: Boolean,
    val data: T? = null,
    val message: String? = null,
    val errors: List<String>? = null
) {
    companion object {
        fun <T> success(data: T, message: String? = null) =
            ApiResponse(success = true, data = data, message = message)

        fun <T> error(message: String, errors: List<String>? = null) =
            ApiResponse<T>(success = false, message = message, errors = errors)
    }
}

ApiResponse<T> é um wrapper genérico que padroniza todas as respostas da API. Independente de ser login, listagem ou erro, o cliente sempre recebe o mesmo envelope:

SituaçãoJSON retornado
Sucesso com dado {"success": true, "data": {...}, "message": "OK"}
Erro de negócio {"success": false, "message": "Credenciais inválidas"}
Erro de validação {"success": false, "message": "Dados inválidos", "errors": ["Campo nome é obrigatório"]}

PasswordUtils

utils/PasswordUtils.ktobject PasswordUtils {
    fun hashPassword(plain: String): String =
        BCrypt.hashpw(plain, BCrypt.gensalt(12))

    fun verifyPassword(plain: String, hashed: String): Boolean =
        BCrypt.checkpw(plain, hashed)
}

BCrypt é um algoritmo de hash especial para senhas. Diferente de MD5 ou SHA, ele é propositalmente lento e inclui salt automático, tornando ataques de força bruta impraticáveis. O número 12 é o "fator de custo".

🚫 Nunca armazene senhas em texto puro! Sempre use PasswordUtils.hashPassword() ao salvar e PasswordUtils.verifyPassword() ao comparar. Nunca compare strings diretamente.

12 Agora Sim: As Rotas

Com toda a infraestrutura montada, uma rota se torna muito simples. Ela apenas recebe a requisição, delega ao Service e retorna a resposta: 

routes/AuthRoutes.kt (simplificado)fun Route.authRoutes() {
    val authService: AuthService by inject()   // Koin entrega a instância

    route("/auth") {
        post("/login") {
            val request = call.receive<LoginRequest>()  // DTO entra
            val tokens  = authService.login(request)    // service faz o trabalho
            call.respond(OK, ApiResponse.success(tokens)) // resposta padronizada
        }

        authenticate("auth-jwt") {        // protege com JWT
            post("/logout") {
                val userId = call.currentUserId()!!  // extensão JWT
                authService.logout(userId)
                call.respond(OK, ApiResponse.success(null, "Logout realizado"))
            }
        }
    }
}

A rota não sabe:

  • onde os dados estão armazenados (responsabilidade do Repository)
  • como o hash de senha funciona (responsabilidade do PasswordUtils via Service)
  • como o JWT é gerado (responsabilidade do JwtConfig via Service)

Ela só sabe: receber, delegar, responder.
