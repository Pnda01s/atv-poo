1 Introdução Técnica

Esta especificação define padrões técnicos, arquitetura, contratos de endpoints, modelos de dados, regras operacionais e mecanismos de comunicação em tempo real da API. A API é implementada em Kotlin com o framework Ktor. O documento abrange tanto a camada REST quanto a camada WebSocket nativa do Ktor.

2 Visão Geral da API

A API FilaCidadã - Em construção segue o padrão REST, utilizando JSON como formato de troca de dados. Todos os recursos são expostos por meio de endpoints acessados via HTTP. Adicionalmente, a API disponibiliza comunicação em tempo real por meio de WebSocket nativo do Ktor para eventos de filas e senhas.

Tecnologias principais:

TecnologiaDescrição
LinguagemKotlin (JVM)
FrameworkKtor (Netty engine)
Banco de dadosMongoDB com KMongo (driver Kotlin para MongoDB)
AutenticaçãoJWT (access token + refresh token) via Ktor Auth
SerializaçãoKotlinx.serialization (schemas de entrada/saída)

DocumentaçãoSwagger/OpenAPI 3.0 via Ktor OpenAPI plugin em /api-docs

3 Arquitetura Técnica da API

A API adota uma arquitetura em camadas com injeção de dependência via Koin, garantindo manutenibilidade, testabilidade e clareza na separação de responsabilidades.

  • Camada de rotas (Routing) — mapeamento de endpoints HTTP via Ktor Routing DSL, aplicação de plugins (autenticação, autorização, validação) e encaminhamento para os controladores.
  • Camada de controladores — extração de dados da requisição (call.receive(), call.parameters) e encaminhamento para a camada de serviços.
  • Camada de serviços — regras de negócio: validações de permissões, prioridades, controle de duplicidade, isolamento institucional e emissão de eventos WebSocket.
  • Camada de repositórios — acesso aos dados persistidos via KMongo, abstraindo o banco de dados MongoDB.
  • Componentes acessórios — autenticação JWT (Ktor Auth), autorização por papéis (plugin customizado), auditoria, tratamento centralizado de erros (StatusPages plugin), validação de entrada, upload de arquivos e integração com serviço de e-mail.

4 Convenções e Padrões Técnicos

Todos os endpoints seguem convenções consistentes de nomenclatura, utilizando substantivos no plural em português brasileiro. Verbos HTTP são empregados de acordo com sua semântica (GET para leitura, POST para criação, PATCH para atualização parcial, DELETE para remoção).

As respostas utilizam JSON como formato padrão. Datas são representadas no formato ISO 8601.

Listagens utilizam paginação padronizada com os parâmetros page e limit, retornando metadados conforme descrito no Apêndice C.

Erros são retornados com códigos HTTP apropriados e corpo JSON contendo mensagem descritiva.

5 Autenticação e Sessão

A autenticação é baseada em dois tokens JWT:

  • Access Token — token de curta duração enviado no cabeçalho Authorization: Bearer {token}. Payload: { id, papeis, instituicaoId }.
  • Refresh Token — token de longa duração utilizado para renovar o access token sem nova autenticação.

Fluxo de autenticação:

  1. O cliente envia credenciais via POST /auth/login.
  2. A API valida credenciais e status do usuário, retornando { token, refresh, expiraEm }.
  3. O access token é enviado em todas as requisições protegidas via cabeçalho Authorization: Bearer.
  4. Quando o access token expira (resposta 401), o cliente utiliza POST /auth/refresh com o refresh token para obter novos tokens.
  5. O logout é realizado via POST /auth/logout, que invalida os tokens no servidor.

Recuperação de senha:

  1. O cliente solicita recuperação via POST /auth/recover informando o e-mail.
  2. A API envia e-mail contendo um link com token único e um código de 4 caracteres.
  3. A redefinição pode ser feita por token (PATCH /auth/password/reset/{token}) ou por código (PATCH /auth/password/reset/code).

Introspecção: O endpoint POST /auth/introspect permite verificar a validade de um access token, retornando informações sobre expiração e tipo.

6 Autorização e Isolamento Institucional

A autorização determina se um usuário autenticado possui permissão para executar determinada ação, considerando o papel atribuído e o contexto institucional.

Papéis do sistema:

PapelDescrição
ADMIN_PLATAFORMAAdministrador global da plataforma. Gerencia instituições, usuários e tem visão total do sistema. Não atua operacionalmente em filas.
ADMIN_INSTITUICAOAdministrador de uma instituição específica. Gerencia filas, usuários da instituição, auditoria e QR codes. Pode atuar no atendimento.
OPERADORUsuário operacional vinculado a uma instituição. Atua no atendimento de filas (chamar, finalizar) e pode gerenciar filas e QR codes.
USUARIO_FINALCidadão que utiliza o sistema. Cria senhas em filas, consulta status, cancela senhas próprias e gerencia perfil.

Regras de isolamento:

  • ADMIN_INSTITUICAO e OPERADOR possuem vínculo obrigatório com uma instituição (instituicaoId). Todas as operações são restritas ao escopo da instituição vinculada.
  • USUARIO_FINAL não possui vínculo institucional e interage apenas com recursos públicos ou com dados de suas próprias senhas.
  • ADMIN_PLATAFORMA tem visão global mas não opera em filas diretamente.
  • Um usuário pode possuir múltiplos papéis (campo papeis é um array).

A matriz completa de permissões por endpoint está no Apêndice B.

8 Regras de Negócio Operacionais

As regras de negócio operacionais definem o comportamento esperado da API. São aplicadas exclusivamente na camada de serviços.

8.1 Criação de Senhas

O fluxo para criação de uma senha segue, obrigatoriamente, a seguinte ordem:

  1. Verificar se o usuário está autenticado e ativo no sistema.
  2. Verificar se a fila está ativa e apta a receber novas senhas.
  3. Verificar se o usuário já possui uma senha ativa (AGUARDANDO ou EM_ATENDIMENTO) para a mesma fila. Caso exista, a criação é negada (409).
  4. Avaliar se a fila exige validação de QR Code (filas presenciais/híbridas).
  5. Aplicar regras de prioridade, quando configuradas.
  6. Calcular a posição da senha na fila, considerando a ordenação vigente.
  7. Persistir a nova senha com status AGUARDANDO.
  8. Emitir evento WebSocket senha:criada para os rooms relevantes.

Em caso de falha em qualquer etapa, a criação é interrompida sem efeitos colaterais parciais.

8.2 Filas Presenciais e QR Code

Para filas com tipoAtendimento PRESENCIAL ou HIBRIDO, a criação de senhas requer validação de um QR Code. Um QR Code é considerado válido quando:

  1. Está associado à fila correta (filaId corresponde).
  2. Encontra-se dentro do período de validade principal (validoAte) ou dentro da janela de tolerância (toleranciaAte).
  3. Está ativo (ativo: true) — não foi invalidado por expiração ou substituição.

A janela de tolerância permite que códigos recém-expirados sejam aceitos por um período adicional limitado, evitando prejuízo ao usuário em cenários de troca recente do QR Code.

A validação ocorre exclusivamente no servidor. Códigos inválidos, expirados ou fora da janela resultam na recusa da criação da senha (409).

8.3 Prioridades

Quando a fila possui prioridadesHabilitadas: true, a API aplica a seguinte hierarquia de pesos para ordenação:

PesoTipo de PrioridadeDescrição
1 (maior)LEGALPrioridade legal (idoso 60+, PCD, gestante)
2FIDELIDADEPrioridade por critérios institucionais
3 (menor)CRONOLÓGICAOrdem de chegada (padrão)

Em caso de empate no nível de prioridade, a ordenação é determinada pela data de criação (createdAt). As prioridades influenciam a posição inicial da senha no momento da criação, sem reordenação retroativa.

8.4 Cancelamento e Encerramento

Cancelamento:

  • Permitido apenas quando o status é AGUARDANDO. Senhas em EM_ATENDIMENTO não podem ser canceladas.
  • Pode ser realizado pelo próprio usuário (USUARIO_FINAL) ou por usuários administrativos.
  • Altera o status para CANCELADA.

Chamar senha:

  • Altera o status de AGUARDANDO para EM_ATENDIMENTO.
  • Permitido apenas para ADMIN_INSTITUICAO e OPERADOR.
  • Emite evento WebSocket senha:chamada.

Finalizar atendimento:

  • Altera o status de EM_ATENDIMENTO para FINALIZADA.
  • Permitido apenas para ADMIN_INSTITUICAO e OPERADOR.
  • Libera o usuário para nova solicitação na mesma fila.
  • Emite evento WebSocket senha:finalizada.

Tanto o cancelamento quanto o encerramento geram registros de auditoria. As transições de estado estão detalhadas no Apêndice A.

8.5 Criação de Usuários (Password-less)

Usuários institucionais (ADMIN_INSTITUICAO, OPERADOR) são criados sem senha pelo administrador:

  1. O administrador cria o usuário via POST /instituicoes/{id}/usuarios fornecendo apenas nome, email e papeis.
  2. A API gera um token único e um código de recuperação.
  3. Um e-mail de boas-vindas é enviado ao novo usuário contendo link e código para definir sua senha.
  4. O usuário acessa o link ou informa o código para definir sua senha pela primeira vez.

9 Contrato de Endpoints

Este capítulo define o contrato técnico entre a API e as aplicações clientes. Todos os 57 endpoints estão documentados abaixo, organizados por domínio.

Todos os endpoints protegidos exigem autenticação via Authorization: Bearer {token}. A autorização é avaliada na camada de serviços. Respostas utilizam JSON. Listagens seguem o padrão do Apêndice C.

9.1 Autenticação

POST /auth/login 🔓 público

Autentica o usuário e emite access token + refresh token. Valida credenciais, verifica status (ativo), gera tokens JWT com payload { id, papeis, instituicaoId }, registra ultimoLoginEm.

"email": "string",
"senha": "string"

Resposta 200: { token, refresh, expiraEm }

Erros: 400 · 401 · 403

POST /auth/register 🔓 público

Registra um novo usuário final (auto-cadastro). O usuário é criado com papel USUARIO_FINAL.

"nome": "string (required)",
"email": "string (required, formato e-mail)",
"senha": "string (required, mín. 8 caracteres)"

Resposta 201: { token, refresh, expiraEm }

Erros: 400 · 409

POST /auth/refresh 🔓 público

Renova access token e refresh token a partir de um refresh token válido.

"token": "string (refresh token)"

Resposta 200: { token, refresh, expiraEm }

Erros: 401

POST /auth/logout autenticado

Encerra a sessão, invalidando os tokens no servidor.

Resposta 200: { message: "Logout realizado" }

Erros: 401

POST /auth/introspect 🔓 público

Verifica a validade de um access token, retornando informações sobre expiração e tipo.

"accessToken": "string"

Resposta 200: { active, tokenType, exp, iat }

Erros: 400

9.2 Recuperação de Senha

POST /auth/recover 🔓 público

Solicita recuperação de senha por e-mail. Gera token único (link) e código de 4 caracteres.

"email": "string (formato e-mail)"
Nota: Em ambiente de desenvolvimento (ktor.environment = development), a resposta inclui token e codigo para facilitar testes.

Erros: 400

PATCH /auth/password/reset/{token} 🔓 público

Redefine a senha utilizando o token recebido por e-mail (via link). O token é informado diretamente na URL como parâmetro de rota.

"senha": "string (nova senha)"

Erros: 400 · 401

PATCH /auth/password/reset/code 🔓 público

Redefine a senha utilizando o código de 4 caracteres recebido por e-mail.

"codigo": "string (4 chars)",
"senha": "string (nova senha)"

Erros: 400 · 401

11 Especificação Técnica de QR Code

O mecanismo de QR Code é utilizado como validação de presença física em filas presenciais. Impede reutilização indevida, compartilhamento remoto e tentativas de fraude.

Regras de geração:

  • Cada QR Code possui código aleatório, tempo de validade limitado (validoAte) e janela de tolerância (toleranciaAte).
  • Geração e rotação ocorrem em intervalos regulares definidos pela configuração da fila (configuracaoQRCode.tempoExibicaoMin).
  • QR Codes não devem conter informações sensíveis ou identificadores de usuários em formato legível.

Regras de validação:

  • Validação exclusivamente no servidor.
  • Verificação de: associação com a fila correta, período de validade, janela de tolerância e status ativo.
  • Códigos inválidos resultam em erro 409.

Endpoints CRUD: Descritos no §9.7 (GET, POST, GET/:id, PATCH desativar).

12 Auditoria e Logs

O sistema de auditoria registra ações administrativas relevantes, permitindo rastreabilidade completa. Os registros são imutáveis e vinculados à instituição.

Eventos auditados:

  • Login de usuários
  • Criação, alteração e remoção de instituições, filas, senhas, usuários e QR codes
  • Cancelamento e finalização de atendimentos
  • Alterações de perfil

Consulta: O endpoint GET /auditorias permite filtragem por instituicaoId, usuarioId, acao, com paginação (ver §9.8).

13 Segurança Técnica

  • Autenticação JWT com access token + refresh token via Ktor Auth plugin e invalidação no servidor via logout
  • Autorização por papéis com plugin customizado e isolamento institucional
  • Validação de entrada com Kotlinx.serialization e validação manual em data classes em todos os endpoints
  • Tokens de recuperação com expiração (token único + código de 4 caracteres)
  • Senhas hasheadas com BCrypt (jBCrypt — nunca armazenadas em texto puro)
  • Upload seguro com validação de tipo MIME e tamanho
  • Campos sensíveis excluídos da serialização via @Transient ou projeções na camada de repositório

Todas as decisões de permissões e regras de negócio são realizadas no servidor. O cliente é considerado ambiente não confiável.

14 Integração com Aplicações Cliente

As aplicações clientes devem tratar a API como fonte única de verdade. Nenhuma decisão de autorização ou regra de negócio deve ser implementada no cliente.

Fluxo de integração:

  1. Autenticação: POST /auth/login → armazenar token e refresh localmente.
  2. Requisições protegidas: enviar Authorization: Bearer {token} em todas as requisições.
  3. Refresh automático: ao receber 401, utilizar POST /auth/refresh. Se falhar, redirecionar para login.
  4. WebSocket: conectar ao endpoint /ws?token={jwt} e fazer join nos rooms relevantes via mensagem JSON.
  5. Logout: POST /auth/logout + limpar tokens locais + desconectar WebSocket.

A APÊNDICE A – Estados da Senha e Transições Permitidas

A.1 Estados Possíveis da Senha

AGUARDANDO
→
EM_ATENDIMENTO
→
FINALIZADA
AGUARDANDO
→
CANCELADA
  • AGUARDANDO — Estado inicial após criação. O usuário aguarda ser chamado.
  • EM_ATENDIMENTO — Quando a senha é chamada. Não pode ser cancelada.
  • CANCELADA — Senha cancelada. Estado terminal.
  • FINALIZADA — Atendimento concluído. O usuário é liberado para nova senha. Estado terminal.

A.2 Transições de Estado Permitidas

Estado AtualPróximo EstadoAçãoPermissão
AGUARDANDOEM_ATENDIMENTOPOST /senhas/{id}/chamarAI, OP
AGUARDANDOCANCELADAPOST /senhas/{id}/cancelarTodos (UF: própria)
EM_ATENDIMENTOFINALIZADAPOST /senhas/{id}/finalizarAI, OP

A.3 Transições Não Permitidas

  • CANCELADA → qualquer outro estado
  • FINALIZADA → qualquer outro estado
  • EM_ATENDIMENTO → CANCELADA
  • FINALIZADA → AGUARDANDO
  • CANCELADA → AGUARDANDO

Qualquer tentativa de transição inválida resulta em erro 409.

A.4 Regras Complementares

  • O estado é atualizado exclusivamente no servidor.
  • O cliente não deve inferir ou alterar estados de forma autônoma.
  • Estados devem ser representados por valores controlados (enumeração) — nunca strings livres.
  • Alterações de estado devem ser atômicas para evitar inconsistências.
  • Toda transição gera registro de auditoria.
  • Toda transição emite evento WebSocket correspondente.
