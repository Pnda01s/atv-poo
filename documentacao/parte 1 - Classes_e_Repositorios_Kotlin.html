1 O que é uma Classe?

Uma classe é o molde (ou planta baixa) a partir do qual criamos objetos. Ela define quais atributos (dados) e quais comportamentos (funções) os objetos desse tipo terão.

💡 Conceito fundamental — OOP Orientação a Objetos é um paradigma de programação que organiza o código em objetos, que combinam estado (atributos) e comportamento (métodos). Uma classe é a descrição desse objeto; o objeto em si é chamado de instância.

1.1 Classes vs. Objetos

A distinção entre classe e objeto é fundamental:

ConceitoAnalogia do mundo realNo código
ClasseA planta baixa de uma casaDefinição em class Usuario { ... }
Objeto (instância)A casa construída a partir da plantaval u = Usuario(...)
AtributoO número de quartos, a metragemval nome: String
MétodoAbrir a porta, ligar a luzfun ativar() { ... }

1.2 Anatomia de uma classe Kotlin

Vamos partir de uma classe simples e ir evoluindo:

exemploclass Pessoa {
    // atributos
    var nome: String = ""
    var idade: Int = 0

    // método
    fun apresentar() {
        println("Olá, meu nome é $nome e tenho $idade anos.")
    }
}

Usando a classe:

usoval p = Pessoa()    // cria uma instância
p.nome = "Ana"     // define o atributo
p.idade = 20
p.apresentar()      // → "Olá, meu nome é Ana e tenho 20 anos."
⚠️ Observação: mutabilidade Em Kotlin, var declara uma variável mutável (pode ser reatribuída) e val declara uma variável imutável (somente leitura, como final em Java). Prefira val sempre que possível — favorece a previsibilidade e é compatível com dados imutáveis.

2 Data Classes

Uma data class é uma classe especial do Kotlin projetada para armazenar dados. Ela elimina todo o código repetitivo (boilerplate) que normalmente teríamos que escrever manualmente.

2.1 O que muda com data class?

Sem data class
class Pessoa(
    val nome: String,
    val idade: Int
) {
    // precisa implementar manualmente:
    override fun toString() = "Pessoa(nome=$nome)"
    override fun equals(o: Any?) = ...
    override fun hashCode() = ...
    fun copy(...) = ...
}
Com data class
data class Pessoa(
    val nome: String,
    val idade: Int
)

// toString(), equals(), hashCode()
// e copy() já vêm de graça!

O compilador Kotlin gera automaticamente, para toda data class:

  • toString() — representação legível: Pessoa(nome=Ana, idade=20)
  • equals() — compara valores (não endereços de memória)
  • hashCode() — necessário para uso em coleções como Set e Map
  • copy() — cria uma cópia com campos opcionalmente alterados
  • componentN() — suporte a desestruturação: val (nome, idade) = pessoa

2.2 Construtores primário e secundário

Em Kotlin, o construtor primário é declarado diretamente no cabeçalho da classe. Ele define ao mesmo tempo os parâmetros e as propriedades:

data class Pessoa(
    val nome: String,    // propriedade + parâmetro
    val idade: Int,
    val email: String
)

Para criar uma instância:

val ana = Pessoa(
    nome  = "Ana",
    idade = 20,
    email = "ana@exemplo.com"
)

// usando copy() para criar variante
val clone = ana.copy(nome = "Ana Luísa")
println(clone) // → Pessoa(nome=Ana Luísa, idade=20, email=ana@exemplo.com)

2.3 Valores padrão e nulidade

Kotlin possui um sistema de tipos que distingue referências nulas (nullable) de não nulas. Isso elimina os NullPointerException em tempo de compilação.

TipoAceita null?Exemplo
String❌ Nãoval nome: String = "Ana"
String?✅ Simval apelido: String? = null
Int❌ Nãoval idade: Int = 20
Int?✅ Simval pontos: Int? = null

Podemos combinar tipos nullable com valores padrão para tornar os parâmetros opcionais:

data class Pessoa(
    val nome: String,
    val email: String,
    val ativo: Boolean = true,   // padrão: true
    val apelido: String? = null  // nullable, padrão: null
)

val p1 = Pessoa("Ana", "ana@email.com")              // ativo=true, apelido=null
val p2 = Pessoa("Bob", "bob@email.com", ativo = false) // ativo=false

3 Enumerações — enum class

Um enum (enumeração) é uma classe especial que representa um conjunto finito e conhecido de valores. Em vez de usar strings soltas como "ADMIN" ou "USER" — que podem ter erros de digitação —, usamos uma enumeração que o compilador valida.

3.1 Criando um enum simples

enum class DiaDaSemana {
    SEGUNDA, TERCA, QUARTA, QUINTA, SEXTA, SABADO, DOMINGO
}

// Usando:
val dia = DiaDaSemana.SEXTA
if (dia == DiaDaSemana.SEXTA) println("Já é sexta! 🎉")

// when (equivalente ao switch, mas mais poderoso):
when (dia) {
    DiaDaSemana.SABADO, DiaDaSemana.DOMINGO -> println("Fim de semana")
    else -> println("Dia útil")
}

3.2 Enum com propriedades

Enums podem ter propriedades e métodos como qualquer outra classe:

enum class Prioridade(val peso: Int) {
    LEGAL(1),         // maior prioridade
    FIDELIDADE(2),
    CRONOLOGICA(3);  // menor prioridade

    fun descricao(): String = when (this) {
        LEGAL       -> "Prioridade legal (idoso, PCD, gestante)"
        FIDELIDADE  -> "Prioridade por fidelidade"
        CRONOLOGICA -> "Ordem de chegada"
    }
}

// Usando:
val p = Prioridade.LEGAL
println(p.peso)       // → 1
println(p.descricao()) // → "Prioridade legal (idoso, PCD, gestante)"
✅ Por que usar enum em vez de String?
  • O compilador verifica todos os valores possíveis — o when avisa se você esqueceu um caso.
  • Autocompletar na IDE funciona perfeitamente.
  • Impossível escrever "ADMINISTRADOR" por engano — o tipo é validado em tempo de compilação.

5 Por que Existem os Repositórios?

Imagine que você precisa buscar um usuário pelo e-mail. A lógica mais ingênua seria escrever diretamente a consulta ao banco de dados em qualquer lugar do código onde precisar:

5.1 O problema sem o padrão Repository

// ❌ RUIM: mistura lógica de negócio com acesso ao banco
fun login(email: String, senha: String) {
    val collection = database.getCollection<Usuario>("usuarios")
    val usuario = collection.findOne(Usuario::email eq email)
    // ... validação da senha ...
}

fun buscarPerfil(email: String) {
    val collection = database.getCollection<Usuario>("usuarios")
    val usuario = collection.findOne(Usuario::email eq email)
    // código de banco duplicado!
}
⚠️ Problemas desse design:
  • Duplicação: a consulta ao banco está espalhada pelo código.
  • Acoplamento: a lógica de negócio conhece detalhes do banco (MongoDB, KMongo, nome da coleção).
  • Testabilidade: é impossível testar o serviço sem um banco de dados real.
  • Manutenção: se a estrutura do banco mudar, você precisa alterar vários lugares.

5.2 A arquitetura em camadas

A solução é separar as responsabilidades em camadas. O padrão Repository é uma das formas mais populares de fazer isso:

ℹ️ Routes vs. Controller — qual a diferença prática?
Em frameworks como o Ktor, é comum que o arquivo de routes já contenha o código do controller (o handler da lambda). A separação conceitual existe para clareza: Routes cuida de qual URL ativa qual lógica; Controller cuida de como ler os dados da requisição e montar a resposta. Em projetos maiores, são arquivos separados.

O Service não sabe como os dados são armazenados nem como a resposta HTTP será formatada. Ele apenas chama métodos do Repository, como findByEmail(). Essa separação torna cada camada independentemente testável e substituível.

6 Interface do Repositório

6.1 O que é uma Interface em Kotlin?

Uma interface é um contrato. Ela declara o quê deve ser feito, sem dizer como. Quem assina o contrato (a classe que a implementa) é obrigada a fornecer o "como".

💡 Analogia: contrato de aluguel A interface é como um contrato de aluguel que diz "o inquilino deve pagar aluguel mensalmente e manter o imóvel conservado". O contrato não diz como o inquilino vai arrumar o dinheiro ou como vai conservar o imóvel — isso é decisão dele. Mas ele é obrigado a cumprir o que está no contrato.

Em Kotlin, uma interface é declarada com a palavra-chave interface. Seus métodos não têm corpo (somente a assinatura):

interface Veiculo {
    fun acelerar(velocidade: Int)
    fun frear()
    fun ligar(): Boolean
}

// Quem quiser ser um "Veículo" deve implementar todos os métodos:
class Carro : Veiculo {
    override fun acelerar(velocidade: Int) { /* implementação específica de carro */ }
    override fun frear()                      { /* ... */ }
    override fun ligar(): Boolean             { return true }
}

6.2 A interface UsuarioRepository

Veja o código real da interface do projeto:

repository/UsuarioRepository.kt/**
 * Interface do repositório de Usuários.
 * Contrato para operações CRUD e consultas específicas.
 * Cria obrigações de implementação, garantindo consistência e flexibilidade.
 */
interface UsuarioRepository {

    fun findById(id: String): Usuario?

    fun findByEmail(email: String): Usuario?

    fun findAll(
        page: Int,
        limit: Int,
        filters: Map<String, Any?> = emptyMap()
    ): Pair<List<Usuario>, Long>

    fun findByInstituicaoId(
        instituicaoId: String,
        page: Int,
        limit: Int,
        filters: Map<String, Any?> = emptyMap()
    ): Pair<List<Usuario>, Long>

    fun insert(usuario: Usuario): Usuario

    fun update(id: String, updates: Map<String, Any?>): Boolean

    fun delete(id: String): Boolean

    fun findByTokenUnico(token: String): Usuario?

    fun findByCodigoRecuperacao(codigo: String): Usuario?
}

6.3 Análise das assinaturas dos métodos

MétodoRetornoPor que esse retorno?
findById(id) Usuario? Pode não existir um usuário com esse ID — ? indica que o resultado pode ser null.
findByEmail(email) Usuario? Idem — e-mail pode não estar cadastrado.
findAll(page, limit, filters) Pair<List<Usuario>, Long> Retorna dois valores: a lista de usuários da página atual e o total de registros (para paginação).
insert(usuario) Usuario Retorna o usuário com o id gerado pelo banco preenchido.
update(id, updates) Boolean true se o documento foi encontrado e modificado; false se nenhum documento foi alterado.
delete(id) Boolean true se o documento foi encontrado e removido; false caso contrário.
ℹ️ O tipo Pair<A, B>
Pair é uma classe genérica do Kotlin que agrega dois valores de tipos possivelmente diferentes. Pair<List<Usuario>, Long> retorna ao mesmo tempo a lista paginada de usuários (.first) e a contagem total (.second). Isso evita ter que criar uma classe extra só para transportar esses dois dados.

7 Implementação do Repositório

A interface define o contrato. A implementação define o como. No projeto FilaCidadã, usamos o banco MongoDB com o driver KMongo para Kotlin.

7.1 Código completo — UsuarioRepositoryImpl

repository/UsuarioRepositoryImpl.ktclass UsuarioRepositoryImpl(
    private val collection: MongoCollection<Usuario>   // (A)
) : UsuarioRepository {

    override fun findById(id: String): Usuario? {
        return collection.findOneById(id)                  // (B)
    }

    override fun findByEmail(email: String): Usuario? {
        return collection.findOne(Usuario::email eq email.lowercase())
    }

    override fun findAll(
        page: Int, limit: Int, filters: Map<String, Any?>
    ): Pair<List<Usuario>, Long> {
        val bsonFilters = buildFilters(filters)
        val total = collection.countDocuments(and(bsonFilters))
        val docs  = collection.find(and(bsonFilters))
            .skip((page - 1) * limit)
            .limit(limit)
            .toList()
        return Pair(docs, total)
    }

    override fun findByInstituicaoId(
        instituicaoId: String, page: Int, limit: Int,
        filters: Map<String, Any?>
    ): Pair<List<Usuario>, Long> {
        val allFilters = filters + ("instituicaoId" to instituicaoId) // (C)
        return findAll(page, limit, allFilters)
    }

    override fun insert(usuario: Usuario): Usuario {
        collection.insertOne(usuario)
        return usuario
    }

    override fun update(id: String, updates: Map<String, Any?>): Boolean {
        val setUpdates = updates.map { (key, value) -> Updates.set(key, value) } // (D)
        val result = collection.updateOneById(id, combine(setUpdates))
        return result.modifiedCount > 0
    }

    override fun delete(id: String): Boolean {
        return collection.deleteOneById(id).deletedCount > 0
    }

    override fun findByTokenUnico(token: String): Usuario? {
        return collection.findOne(Usuario::tokenUnico eq token)
    }

    override fun findByCodigoRecuperacao(codigo: String): Usuario? {
        return collection.findOne(Usuario::codigoRecuperaSenha eq codigo)
    }

    // ── método privado auxiliar ──────────────────────────────
    private fun buildFilters(filters: Map<String, Any?>): List<Bson> {  // (E)
        return filters.mapNotNull { (key, value) ->
            when {
                value == null        -> null
                key == "nome"        -> Usuario::nome regex Regex(".*$value.*", RegexOption.IGNORE_CASE)
                key == "email"       -> Usuario::email regex Regex(".*$value.*", RegexOption.IGNORE_CASE)
                key == "ativo"       -> Usuario::ativo eq (value as Boolean)
                key == "instituicaoId" -> Usuario::instituicaoId eq (value as String)
                else                -> null
            }
        }
    }
}

7.2 Entendendo método a método

(A) Injeção da coleção via construtor

A classe recebe a coleção MongoDB no construtor (MongoCollection<Usuario>). Isso é chamado de injeção de dependência — a classe não cria sua dependência, ela recebe de fora. Isso facilita os testes (é possível passar uma coleção em memória).

// O construtor diz: "me dê a coleção, eu farei o resto"
class UsuarioRepositoryImpl(private val collection: MongoCollection<Usuario>)

(B) Operadores infix do KMongo — DSL de consulta

O KMongo oferece uma DSL (sintaxe de domínio específico) para construir filtros MongoDB de forma tipada em Kotlin. Em vez de strings com o nome do campo (que podem ter erros de digitação), usamos referências reais à propriedade:

// ❌ Sem type-safety: string pode estar errada
collection.find("{ email: '$email' }")

// ✅ Com KMongo DSL: compilador valida o nome do campo
collection.findOne(Usuario::email eq email)
// Usuario::email → referência à propriedade (KProperty)
// eq           → operador "igual" (infix)

Se você renomear o campo email na classe Usuario, o compilador imediatamente apontará o erro aqui.

(C) Paginação com skip + limit

A paginação funciona assim: para obter a página N com L itens por página, pulamos os primeiros (N - 1) × L documentos e buscamos os próximos L:

page = 3, limit = 10
skip = (3 - 1) × 10 = 20  // pula os primeiros 20
limit = 10                 // pega os próximos 10 (documentos 21 a 30)
collection.find(filtro)
    .skip((page - 1) * limit)
    .limit(limit)
    .toList()

(D) Atualização parcial com Updates.set

O método update recebe um Map<String, Any?> com apenas os campos a alterar. Ele converte cada entrada em um comando $set do MongoDB:

// Chamada: atualizar só o nome e o status
repository.update(id, mapOf("nome" to "Carlos", "ativo" to false))

// Internamente gera: { $set: { nome: "Carlos", ativo: false } }
val setUpdates = updates.map { (key, value) -> Updates.set(key, value) }
collection.updateOneById(id, combine(setUpdates))

O MongoDB atualiza somente os campos informados, sem tocar nos outros.

7.3 O método buildFilters — construindo filtros dinamicamente

Um dos pontos mais elegantes da implementação é o método privado buildFilters. Ele recebe um mapa genérico de filtros e constrói filtros tipados do MongoDB:

private fun buildFilters(filters: Map<String, Any?>): List<Bson> {
    return filters.mapNotNull { (key, value) ->
        when {
            value == null          -> null  // filtro não aplicado
            key == "nome"          -> Usuario::nome  regex Regex(".*$value.*", RegexOption.IGNORE_CASE)
            key == "email"         -> Usuario::email regex Regex(".*$value.*", RegexOption.IGNORE_CASE)
            key == "ativo"         -> Usuario::ativo eq (value as Boolean)
            key == "instituicaoId" -> Usuario::instituicaoId eq (value as String)
            else                   -> null  // chave desconhecida, ignora
        }
    }
}

O que acontece passo a passo:

  1. Para cada entrada (key, value) do mapa, aplica a expressão when.
  2. Se value == null, retorna null (o filtro não é aplicado — mapNotNull descarta nulls automaticamente).
  3. Para nome e email, usa uma busca por expressão regular case-insensitive (busca parcial).
  4. Para ativo, faz uma comparação exata de booleano.
  5. A lista resultante é combinada com and(bsonFilters) — todos os filtros ativados são aplicados simultaneamente.
✅ Usar mapNotNull em vez de filter + map
mapNotNull { ... } é equivalente a map { ... }.filterNotNull(), mas em uma única passagem. Ele transforma e filcula em uma operação só — preferível por ser mais conciso e legível.

"Módulos de alto nível não devem depender de módulos de baixo nível. Ambos devem depender de abstrações."

9 Injeção de Dependência com Koin

Como o Service recebe a implementação correta do repositório? Por meio de um container de injeção de dependência. No projeto FilaCidadã, usamos o Koin.

O Koin é um framework leve de injeção de dependência para Kotlin. Você define um módulo que diz ao framework como criar cada objeto:

config/KoinModules.kt (simplificado)val appModule = module {

    // 1. Cria a coleção do MongoDB
    single {
        get<MongoDatabase>().getCollection<Usuario>("usuarios")
    }

    // 2. Vincula a INTERFACE à IMPLEMENTAÇÃO
    single<UsuarioRepository> {
        UsuarioRepositoryImpl(get()) // "get()" injeta a coleção do passo 1
    }

    // 3. Service recebe a interface (não sabe qual implementação)
    single { UsuarioService(get<UsuarioRepository>()) }
}

No service, você simplesmente declara:

services/UsuarioService.kt (simplificado)class UsuarioService(
    private val usuarioRepository: UsuarioRepository  // ← interface, não impl
) {
    fun buscarPorId(id: String): Usuario {
        return usuarioRepository.findById(id)
            ?: throw ApiException(404, "Usuário não encontrado")
    }
}
ℹ️ Por que single { }?
No Koin, single { } cria um singleton — uma única instância compartilhada por toda a aplicação. Para repositórios e serviços, isso é o comportamento desejado: criamos uma vez e reutilizamos.

7a. Estrutura composta — raiz + classes aninhadas

🎯 Resumo do que você aprendeu
  • data class é ideal para modelar dados — gera automaticamente equals, toString e copy.
  • enum class representa um conjunto fixo de valores, com type-safety garantido pelo compilador.
  • val é imutável; var é mutável. Prefira val.
  • String? é diferente de String — o ? indica que o campo pode ser null.
  • Uma interface declara um contrato (o quê). A implementação define como ele é cumprido.
  • O padrão Repository separa o acesso a dados da lógica de negócio.
  • Injeção de dependência (Koin) conecta interfaces às suas implementações na inicialização da aplicação.
  • O Service depende da interface, não da implementação — isso é o Princípio D do SOLID.
