1 Onde Ficam as Regras de Negócio?

Na Parte 1, aprendemos a criar classes (modelos de dados) e repositórios (acesso ao banco). Agora surge a pergunta: onde colocamos a lógica que decide o que pode ou não acontecer no sistema?

Por exemplo: ao criar um usuário, quem valida se o e-mail já existe? Quem faz o hash da senha? Quem dispara o e-mail de boas-vindas? Essas são regras de negócio, e elas precisam de uma casa própria.

1.1 O problema: regras espalhadas pelo código

Considere duas abordagens erradas para cadastrar um usuário:

Regra no Repository
// ❌ ERRADO — Repository não deve validar regras
class UsuarioRepositoryImpl(...) : UsuarioRepository {
    override fun insert(u: Usuario): Usuario {
        // Validação aqui? NÃO deveria!
        if (findByEmail(u.email) != null)
            throw Exception("Email já existe")
        collection.insertOne(u)
        return u
    }
}
Regra na Route
// ❌ ERRADO — Route não deve conter lógica
post("/usuarios") {
    val req = call.receive<CreateRequest>()
    // Validação aqui? NÃO deveria!
    if (repo.findByEmail(req.email) != null)
        throw ApiException(400, "Já existe")
    val hash = BCrypt.hashpw(req.senha, ...)
    val usuario = Usuario(nome = req.nome, ...)
    repo.insert(usuario)
}
⚠️ Por que isso é problemático?
  • Duplicação: se duas rotas precisam criar usuário, a validação se duplica.
  • Responsabilidade misturada: o Repository deveria apenas salvar; a Route deveria apenas receber/devolver HTTP.
  • Dificuldade de teste: para testar a regra de "e-mail único", você precisa de HTTP ou banco real.
  • Manutenção: se a regra mudar (ex.: "e-mail único por instituição"), quantos lugares você precisa alterar?

1.2 O que é um Service?

Um Service é uma classe que encapsula regras de negócio e orquestra operações entre diferentes componentes (repositórios, outros services, serviços externos). Ele é o "cérebro" da aplicação.

💡 Analogia: o gerente do restaurante
  • O garçom (Route) recebe o pedido do cliente e entrega o prato pronto.
  • O gerente (Service) confere se o pedido é válido, verifica estoque, coordena a preparação.
  • O cozinheiro (Repository) prepara e armazena os ingredientes.
O garçom nunca vai à cozinha. O cozinheiro nunca fala com o cliente. O gerente é o intermediário que conhece as regras do restaurante.

Na prática, o Service:

  • Valida dados — e-mail único, senha forte, fila ativa
  • Transforma dados — converte DTO → entidade, faz hash de senha
  • Orquestra — chama um ou mais Repositories e outros Services
  • Decide — lança exceção se algo violar as regras

1.3 O fluxo completo de uma requisição

Reveja o diagrama de camadas da Parte 1, §5.2. Na Parte 1, focamos na camada Repository. Agora, o foco é a camada Service:

🌐 Route
Recebe request HTTP
→
⚙️ Service
Valida, transforma, orquestra
→
📋 Repository
Persiste no banco
→
🗄️ MongoDB

Fluxo detalhado de POST /usuarios:

  1. A Route recebe o JSON, deserializa para um DTO (CreateUsuarioInstituicaoRequest) e chama o Service.
  2. O Service verifica se o e-mail já está cadastrado, valida os papéis, faz o hash da senha, monta a entidade Usuario e chama o Repository.
  3. O Repository insere o documento no MongoDB e retorna o objeto persistido.
  4. O Service registra auditoria, envia e-mail de boas-vindas e retorna o usuário à Route.
  5. A Route responde com 201 Created e o JSON do usuário criado.
ℹ️ Separação de preocupações
A Route não sabe como a validação é feita. O Repository não sabe por quê está inserindo. O Service não sabe qual método HTTP disparou a operação. Cada camada tem uma responsabilidade, e só depende da camada abaixo por meio de abstrações (interfaces).

2 Encapsulamento

2.1 Conceito formal

Encapsulamento é o princípio de OO que consiste em esconder os detalhes internos de uma classe e expor apenas o necessário. Quem usa a classe interage por uma interface pública simples, sem conhecer a implementação interna.

💡 Analogia: o carro
Você dirige um carro usando volante, pedais e câmbio (interface pública). Você não precisa saber como funciona a injeção eletrônica, o ABS ou a embreagem por dentro (detalhes internos). O fabricante pode trocar o motor inteiro e, enquanto volante e pedais funcionarem igual, você não percebe diferença.

No contexto de OO, encapsulamento implica:

  • Atributos internos são privados — o mundo externo não acessa diretamente.
  • O acesso acontece por métodos controlados — garantindo validações e invariantes.
  • Detalhes de implementação podem mudar — sem afetar quem usa a classe.

2.2 Encapsulamento na prática: o UsuarioService

Veja como o UsuarioService encapsula toda a complexidade de criar um usuário vinculado a uma instituição:

services/UsuarioService.kt (simplificado)class UsuarioService(
    private val usuarioRepository: UsuarioRepository,   // (1) private
    private val auditoriaService: AuditoriaService,     // (1) private
    private val emailService: EmailService              // (1) private
) {

    // ── Método PÚBLICO — interface limpa ──────────────
    fun criarParaInstituicao(
        request: CreateUsuarioInstituicaoRequest,
        instituicaoId: String,
        criadorId: String
    ): Usuario {
        validarEmailUnico(request.email)          // (2) privado
        val papeis = validarPapeis(request.papeis)  // (2) privado
        val usuario = montarEntidade(request, papeis, instituicaoId) // (2)

        val criado = usuarioRepository.insert(usuario)
        auditoriaService.registrar(AcaoAuditoria.CRIAR, "Usuario", criado.id, criadorId)
        enviarEmailBoasVindas(criado)             // (2) privado

        return criado
    }

    // ── Métodos PRIVADOS — detalhes escondidos ────────
    private fun validarEmailUnico(email: String) {          // (3)
        if (usuarioRepository.findByEmail(email) != null)
            throw ApiException(400, "E-mail já cadastrado")
    }

    private fun validarPapeis(papeis: List<String>): Set<Papel> {
        return papeis.map {
            try { Papel.valueOf(it) }
            catch (_: Exception) { throw ApiException(400, "Papel inválido: $it") }
        }.toSet()
    }

    private fun montarEntidade(...): Usuario { ... }
    private fun enviarEmailBoasVindas(u: Usuario) { ... }
}

O que o código acima demonstra:

(1) Dependências são private

Ninguém de fora do Service acessa usuarioRepository, auditoriaService ou emailService. A Route não sabe que o Service usa um EmailService — ela só chama criarParaInstituicao().

(2) Métodos privados escondem a complexidade

O método público criarParaInstituicao() lê como uma receita de alto nível: "valide o e-mail, valide os papéis, monte a entidade, insira, registre auditoria, envie e-mail". Os detalhes de como cada passo funciona ficam nos métodos privados.

(3) Mudanças internas não afetam o exterior

Se amanhã a regra mudar de "e-mail único global" para "e-mail único por instituição", você altera apenas o método validarEmailUnico(). A Route continua chamando criarParaInstituicao() sem saber que algo mudou.

2.3 Modificadores de visibilidade em Kotlin

Kotlin oferece quatro modificadores de visibilidade:

ModificadorVisível paraUso típico
public (padrão)Qualquer classe, de qualquer lugarMétodos do Service chamados pela Route
privateApenas dentro da própria classeDependências do construtor, métodos auxiliares
protectedDentro da classe e suas subclassesMétodos reutilizáveis em hierarquia de herança
internalDentro do mesmo módulo (projeto)Classes auxiliares do mesmo pacote
✅ Regra prática
Comece tudo como private. Só torne público o que for necessário para o mundo exterior. Isso é encapsulamento aplicado: minimize a superfície de contato entre as classes.

3 Composição

3.1 O Service usa Repositories — não é um Repository

Composição é o princípio de construir classes complexas combinando objetos mais simples. Em vez de herdar comportamento de outra classe, a classe possui referências a outros objetos e delega a eles.

No UsuarioService, a composição acontece naturalmente:

composição no construtorclass UsuarioService(
    private val usuarioRepository: UsuarioRepository,    // TEM um repo de usuários
    private val auditoriaService: AuditoriaService,      // TEM um service de auditoria
    private val emailService: EmailService               // TEM um service de email
)

Compare com uma abordagem incorreta usando herança:

❌ Herança (errado)
// UsuarioService É UM UsuarioRepository?
// Não faz sentido lógico!
class UsuarioService : UsuarioRepositoryImpl() {
    // Herda todas as operações de banco
    // Não pode usar outro repository
    // Acoplamento total
}
✅ Composição (correto)
// UsuarioService TEM UM UsuarioRepository
// Faz sentido: o Service usa o Repository
class UsuarioService(
    private val repo: UsuarioRepository
) {
    // Pode usar QUALQUER implementação
    // Depende da interface, não do impl
}

3.2 Por que composição em vez de herança?

AspectoHerançaComposição
Relação"É um" (is-a)"Tem um" (has-a)
AcoplamentoAlto — herda tudo da classe paiBaixo — usa apenas a interface
FlexibilidadeHerança única em KotlinPode compor quantos quiser
TestabilidadeDifícil isolarPode injetar mocks
Exemploclass Dog : Animal()class UsuarioService(val repo: ...)

Observe como o SenhaService usa composição com múltiplos repositórios e services:

services/SenhaService.kt (construtor)class SenhaService(
    private val senhaRepository: SenhaRepository,     // para persistir senhas
    private val filaRepository: FilaRepository,       // para verificar se fila está ativa
    private val qrCodeRepository: QrCodeRepository,   // para validar QR Code
    private val auditoriaService: AuditoriaService,   // para registrar ações
    private val webSocketManager: WebSocketManager    // para notificar em tempo real
)

Para criar uma senha, o SenhaService precisa: verificar se a fila existe e está ativa (usa FilaRepository), validar o QR Code (usa QrCodeRepository), calcular a posição (usa SenhaRepository), inserir a senha (usa SenhaRepository), registrar auditoria (usa AuditoriaService) e notificar em tempo real (usa WebSocketManager). Seis objetos colaborando — nenhum herdado, todos compostos.

💡 Princípio clássico de OO
"Prefira composição sobre herança" (Favor composition over inheritance — Gang of Four, 1994). Herança é útil quando existe uma relação genuína de "é um" (ex.: Gato é um Animal). Para montar funcionalidade a partir de partes, composição é a escolha certa.

4 DTOs — Separando Entrada e Saída

4.1 Por que não usar a entidade diretamente?

O modelo Usuario tem 16 atributos, incluindo senhaHash, tokens e timestamps internos. Se expusermos a entidade diretamente na API:

  • O cliente receberia a senhaHash na resposta — falha de segurança.
  • O cliente precisaria enviar id, createdAt e updatedAt — campos que são gerados pelo sistema.
  • Qualquer mudança no modelo quebraria a API — acoplamento.

A solução é usar DTOs (Data Transfer Objects): classes separadas que definem exatamente o que entra e o que sai da API.

💡 DTO — Data Transfer Object
Um DTO é uma data class cujo único propósito é transportar dados entre camadas. Ele não contém lógica de negócio, apenas atributos. No FilaCidadã, dividimos em:
dtos/request/ — o que o cliente envia
dtos/response/ — o que o cliente recebe

4.2 Request DTOs — o que o cliente envia

Veja os DTOs de requisição para endpoints de usuários:

dtos/request/UsuarioRequests.kt@Serializable
data class CreateUsuarioInstituicaoRequest(
    val nome: String,               // obrigatório
    val email: String,              // obrigatório
    val papeis: List<String>         // obrigatório
)

@Serializable
data class UpdateUsuarioRequest(
    val nome: String? = null,       // opcional — só atualiza se informado
    val papeis: List<String>? = null,
    val ativo: Boolean? = null,
    val instituicaoId: String? = null
)

Pontos importantes:

AspectoCreateRequestUpdateRequest
Campos obrigatóriosSim — sem valores padrãoTodos opcionais (null)
Campo id❌ Não — gerado pelo Service❌ Não — vem da URL
Campo senhaHash❌ Não — password-less❌ Não — alteração de senha é outro fluxo
Campo createdAt❌ Não — gerado automaticamente❌ Não — imutável
Por que separar?Cada operação tem campos diferentes. O DTO garante que o cliente envia apenas o que faz sentido.
ℹ️ Nullable = atualização parcial (PATCH semântico)
No UpdateUsuarioRequest, todos os campos são String? = null. O Service verifica quais não são null e só atualiza esses. Se o cliente enviar { "nome": "Carlos" }, apenas o nome muda — os outros campos permanecem inalterados. Isso é possível graças ao sistema de nullable do Kotlin.

4.3 Response DTOs — o que o cliente recebe

O projeto FilaCidadã usa um envelope padrão para todas as respostas da API:

dtos/response/ApiResponse.kt@Serializable
data class ApiResponse<T>(
    val success: Boolean,
    val data: T? = null,
    val message: String? = null,
    val errors: List<String>? = null
) {
    companion object {
        fun <T> success(data: T, message: String? = null) = ApiResponse(
            success = true, data = data, message = message
        )

        fun <T> error(message: String, errors: List<String>? = null) = ApiResponse<T>(
            success = false, message = message, errors = errors
        )
    }
}

Análise dos conceitos de OO nesse código:

T Generics — ApiResponse<T>

O <T> é um parâmetro de tipo genérico. Ele permite que ApiResponse transporte qualquer tipo de dado: ApiResponse<Usuario>, ApiResponse<List<Fila>>, ApiResponse<SenhaStatsResponse>. Uma única classe para todas as respostas.

CO Companion Object — fábrica de respostas

O companion object em Kotlin é equivalente a métodos estáticos em Java. Os métodos success() e error() são métodos de fábrica (factory methods) que criam instâncias padronizadas sem expor o construtor diretamente. Isso garante que toda resposta de sucesso tenha success = true e toda resposta de erro tenha success = false.

Exemplo de uso:

// Na Route, após chamar o Service
val usuario = usuarioService.buscarPorId(id)
call.respond(ApiResponse.success(usuario))
// → { "success": true, "data": { "id": "...", "nome": "Ana", ... } }

// Em caso de erro (lançado pelo Service)
// → { "success": false, "message": "Usuário não encontrado" }

Para listagens paginadas, o sistema usa outro DTO genérico:

utils/PaginationUtils.kt@Serializable
data class PaginatedResponse<T>(
    val docs: List<T>,
    val totalDocs: Long,
    val limit: Int,
    val totalPages: Int,
    val page: Int,
    val hasPrevPage: Boolean,
    val hasNextPage: Boolean
)
ℹ️ Generics e reutilização
Tanto ApiResponse<T> quanto PaginatedResponse<T> são genéricos. Isso significa que uma única classe serve para todos os tipos de dados da API. Sem generics, precisaríamos de ApiResponseUsuario, ApiResponseFila, ApiResponseSenha, etc. — código duplicado para cada entidade.

5 Exceções de Negócio

5.1 A classe ApiException

Quando uma regra de negócio é violada (e-mail duplicado, fila inativa, senha inválida), o Service precisa sinalizar o erro. Em Kotlin, isso é feito lançando uma exceção:

plugins/ApiException.ktclass ApiException(
    val statusCode: Int,           // código HTTP: 400, 404, 409...
    override val message: String,  // mensagem para o cliente
    val errors: List<String>? = null
) : RuntimeException(message)      // ← herança!

Conceitos de OO nesse código:

  • Herança: ApiException herda de RuntimeException — portanto, pode ser lançada com throw e capturada com catch.
  • override: o atributo message sobrescreve o message da classe pai Throwable.
  • Atributo próprio: statusCode é específico da nossa exceção — a RuntimeException padrão não tem código HTTP.

O Service usa ApiException para sinalizar erros semânticos:

uso no Service// 404 — recurso não encontrado
fun buscarPorId(id: String): Usuario {
    return usuarioRepository.findById(id)
        ?: throw ApiException(404, "Usuário não encontrado")
}

// 400 — dados inválidos
if (usuarioRepository.findByEmail(request.email) != null)
    throw ApiException(400, "E-mail já cadastrado")

// 409 — conflito de estado
if (senha.status != StatusSenha.AGUARDANDO)
    throw ApiException(409, "Senha não está aguardando")

5.2 Tratamento centralizado — StatusPages

As exceções lançadas pelo Service são capturadas automaticamente pelo plugin StatusPages do Ktor. Ele converte a exceção em uma resposta HTTP padronizada:

plugins/StatusPages.ktfun Application.configureStatusPages() {
    install(StatusPages) {

        // Captura TODAS as ApiException lançadas por qualquer Service
        exception<ApiException> { call, cause ->
            call.respond(
                HttpStatusCode.fromValue(cause.statusCode),
                ApiResponse.error<Any>(cause.message, cause.errors)
            )
        }

        // Captura erros inesperados (proteção genérica)
        exception<Exception> { call, cause ->
            call.respond(
                HttpStatusCode.InternalServerError,
                ApiResponse.error<Any>("Erro interno do servidor")
            )
        }
    }
}

O fluxo completo do erro:

⚙️ Service
throw ApiException(404, ...)
→
🛡️ StatusPages
Captura a exceção
→
📤 Resposta HTTP
404 { success: false, message: "..." }
✅ Benefícios desse design
  • O Service não precisa saber nada sobre HTTP — só lança exceções com mensagens claras.
  • A Route não precisa de try/catch — o StatusPages cuida de tudo.
  • Toda resposta de erro segue o mesmo formato (ApiResponse) — consistência para o frontend.
  • Basta adicionar um novo tipo de exceção e um handler para expandir o sistema.

6 O UsuarioService Completo — Código Real

6.1 Código do projeto

Agora vamos analisar o UsuarioService completo, da mesma forma que analisamos o UsuarioRepositoryImpl na Parte 1:

services/UsuarioService.ktclass UsuarioService(
    private val usuarioRepository: UsuarioRepository,
    private val auditoriaService: AuditoriaService,
    private val emailService: EmailService
) {

    fun listar(
        pagination: PaginationParams,
        filters: Map<String, Any?>
    ): PaginatedResponse<Usuario> {                              // (A)
        val (docs, total) = usuarioRepository.findAll(
            pagination.page, pagination.limit, filters
        )
        return buildPaginatedResponse(docs, total, pagination)
    }

    fun buscarPorId(id: String): Usuario {                         // (B)
        return usuarioRepository.findById(id)
            ?: throw ApiException(404, "Usuário não encontrado")
    }

    fun criarParaInstituicao(                                     // (C)
        request: CreateUsuarioInstituicaoRequest,
        instituicaoId: String,
        criadorId: String
    ): Usuario {
        if (usuarioRepository.findByEmail(request.email) != null)
            throw ApiException(400, "E-mail já cadastrado")

        val papeis = request.papeis.map {
            try { Papel.valueOf(it) }
            catch (_: Exception) { throw ApiException(400, "Papel inválido: $it") }
        }.toSet()

        val usuario = Usuario(
            id = UUID.randomUUID().toString(),
            nome = request.nome,
            email = request.email.lowercase(),
            senhaHash = "",                        // password-less
            papeis = papeis,
            instituicaoId = instituicaoId
        )

        val criado = usuarioRepository.insert(usuario)
        auditoriaService.registrar(                               // (D)
            AcaoAuditoria.CRIAR, "Usuario", criado.id, criadorId, instituicaoId
        )
        emailService.sendWelcomeEmail(criado.email, criado.nome, ...) // (E)

        return criado
    }

    fun atualizar(                                                 // (F)
        id: String,
        request: UpdateUsuarioRequest,
        executorId: String
    ): Boolean {
        buscarPorId(id)                                           // verifica existência

        val updates = mutableMapOf<String, Any?>()
        request.nome?.let { updates["nome"] = it }              // (G)
        request.ativo?.let { updates["ativo"] = it }
        request.instituicaoId?.let { updates["instituicaoId"] = it }
        request.papeis?.let { papeisList ->
            updates["papeis"] = papeisList.map { Papel.valueOf(it) }.toSet()
        }
        updates["updatedAt"] = java.time.Instant.now()

        val result = usuarioRepository.update(id, updates)
        if (result) auditoriaService.registrar(
            AcaoAuditoria.ATUALIZAR, "Usuario", id, executorId
        )
        return result
    }

    fun deletar(id: String, executorId: String): Boolean {       // (H)
        buscarPorId(id)
        val result = usuarioRepository.delete(id)
        if (result) auditoriaService.registrar(
            AcaoAuditoria.DELETAR, "Usuario", id, executorId
        )
        return result
    }
}

6.2 Análise método a método

(A) listar() — delegação pura

O Service recebe parâmetros de paginação já extraídos e delega ao Repository. Em seguida, formata o resultado como PaginatedResponse. A desestruturação val (docs, total) = ... extrai os dois valores do Pair<List, Long> retornado pelo Repository — recurso da data class que vimos na Parte 1.

(B) buscarPorId() — o operador Elvis ?:

O Repository retorna Usuario? (pode ser null). O Service decide o que fazer se for null: lança uma ApiException(404). O operador ?: (Elvis) é lido como: "retorne o valor à esquerda; se for null, execute a expressão à direita".

return findById(id) ?: throw ApiException(404, "Não encontrado")
//     ↑ pode ser null    ↑ se for null, faz isso

Esse método é reutilizado internamente por atualizar() e deletar() para verificar existência antes de agir.

(C) criarParaInstituicao() — o método mais rico

Este método demonstra como o Service orquestra múltiplas responsabilidades:

  1. Valida — e-mail único, papéis válidos
  2. Transforma — DTO → entidade, converte Strings em enums, normaliza e-mail
  3. Persiste — delega ao Repository
  4. Efeitos colaterais — auditoria, envio de e-mail

Repare que o DTO (CreateUsuarioInstituicaoRequest) tem 3 campos, mas a entidade Usuario tem 16. O Service preenche os campos faltantes (id, senhaHash, instituicaoId, etc.).

(D) Orquestrando outros Services

auditoriaService.registrar(...) é um exemplo de composição em ação: o UsuarioService não sabe como a auditoria é armazenada (pode usar MongoDB, pode usar um serviço externo). Ele apenas chama o método público do AuditoriaService.

(E) Integrando serviços externos

emailService.sendWelcomeEmail(...) envia um e-mail de boas-vindas. O UsuarioService não sabe como o e-mail é enviado (SMTP? API externa?). Novamente: composição + encapsulamento.

(F) atualizar() — atualização parcial

O método recebe um UpdateUsuarioRequest onde todos os campos são nullable. Somente os campos não-null são adicionados ao mapa de atualizações.

(G) let { } — operador de escopo

request.nome?.let { updates["nome"] = it } é lido como: "se request.nome não for null, execute o bloco usando o valor como it". É a forma idiomática do Kotlin para lidar com nullables sem if/else.

// Equivalente sem let:
if (request.nome != null) {
    updates["nome"] = request.nome
}

// Com let (idiomático Kotlin):
request.nome?.let { updates["nome"] = it }

(H) deletar() — verificação + auditoria

Antes de deletar, chama buscarPorId(id) que lança 404 se o usuário não existir. Após a deleção bem-sucedida, registra auditoria. Mesmo o método mais simples segue o padrão: verifica → executa → registra.

A composição permite que cada Service cresça independentemente.

8 Registrando Services no Koin

Na Parte 1, §9, vimos como registrar Repositories no Koin. Agora registramos os Services — que dependem dos Repositories:

config/KoinModules.ktval appModule = module {

    // ── Config ──
    single { MongoConfig() }
    single { JwtConfig() }
    single { WebSocketManager() }

    // ── Repositories (Parte 1) ──
    single<UsuarioRepository> { UsuarioRepositoryImpl(get<MongoConfig>().usuarios) }
    single<InstituicaoRepository> { InstituicaoRepositoryImpl(get<MongoConfig>().instituicoes) }
    single<FilaRepository> { FilaRepositoryImpl(get<MongoConfig>().filas) }
    single<SenhaRepository> { SenhaRepositoryImpl(get<MongoConfig>().senhas) }
    single<QrCodeRepository> { QrCodeRepositoryImpl(get<MongoConfig>().qrcodes) }
    single<AuditoriaRepository> { AuditoriaRepositoryImpl(get<MongoConfig>().auditorias) }

    // ── Serviços auxiliares ──
    single { FileStorageService() }
    single { EmailService() }

    // ── Services (NOVO — dependem dos Repositories) ──
    single { AuditoriaService(get()) }                          // (1) get() → AuditoriaRepository
    single { UsuarioService(get(), get(), get()) }              // (2) get() × 3
    single { InstituicaoService(get(), get()) }                 // (3)
    single { FilaService(get(), get(), get(), get()) }          // (4)
    single { SenhaService(get(), get(), get(), get(), get()) }  // (5) get() × 5
}

Análise:

(1) O Koin resolve automaticamente

Quando escrevemos single { AuditoriaService(get()) }, o Koin olha o construtor de AuditoriaService, vê que ele precisa de um AuditoriaRepository, e procura no container uma instância desse tipo. Como registramos single<AuditoriaRepository> { AuditoriaRepositoryImpl(...) }, a implementação correta é injetada automaticamente.

(2) Cadeia de dependências

UsuarioService(get(), get(), get()) precisa de: UsuarioRepository, AuditoriaService e EmailService. O Koin resolve cada get() pelo tipo do parâmetro no construtor. A ordem de registro no módulo não importa — o Koin monta o grafo de dependências e resolve tudo no momento da primeira utilização.

⚠️ Cuidado com dependências circulares
Se ServiceA dependesse de ServiceB e ServiceB dependesse de ServiceA, o Koin entraria em loop infinito. No FilaCidadã, isso não acontece porque a hierarquia é clara: Services dependem de Repositories e de outros Services mais simples (como AuditoriaService), nunca o contrário.

🎯 Resumo do que você aprendeu
  • Service é a camada que contém regras de negócio — entre Route e Repository.
  • Encapsulamento esconde detalhes internos (private) e expõe apenas o necessário (public).
  • Composição constrói funcionalidade combinando objetos — prefira sobre herança.
  • DTOs separam o formato da API do formato do banco — segurança e desacoplamento.
  • ApiException herda de RuntimeException — herança usada de forma pontual e correta.
  • Generics (<T>) permitem reutilizar ApiResponse e PaginatedResponse para qualquer tipo.
  • Companion object fornece métodos de fábrica — padrão Factory Method.
  • O Koin resolve a cadeia de dependências: Repository → Service → Route.
  • O operador Elvis ?: combina com throw para validações concisas com nullables.
  • ?.let { } é a forma idiomática de executar código somente se o valor não for null.
